# AgentForge — Validate Prompt

You are validating consistency between design-contract.md and its companion diagram, then generating design-contract.json on pass.

## Prerequisites
- `design-contract.md` exists in project root
- Diagram file exists in `diagrams/` (format specified in plan.md: .drawio.xml, .eraser, or .svg)
- MCP Server configured with AGENTFORGE_API_KEY (if required)

## Instructions
1. Read `design-contract.md` from the project root
2. Read the diagram file from `diagrams/` (identify format from file extension)
3. Call `validate_contract(md_content, diagram_content, diagram_format)` on the MCP Server
4. Display the results to the developer:
   - For each check (1-16): show ✅ PASS, ❌ BLOCK, or ⚠️ WARN with description
   - Checks 1-12: structural (section completeness, topology, bindings, patterns, NFRs, etc.)
   - Checks 13-16: diagram↔§3 STL consistency (node parity, edge parity, hierarchy, A2A)
5. If ANY BLOCK finding:
   - Display all findings with fix guidance
   - Do NOT write design-contract.json
   - Tell developer: "Fix the blocking findings in design-contract.md or the diagram, then re-run /validate."
6. If ALL checks pass:
   - Write `design-contract.json` to project root (generated by validate_contract from .md)
   - If attestation is returned, include it in the report
   - Tell developer: "✅ All 16 checks passed. design-contract.json generated. Run /implement to generate code."

## MCP not configured
If the MCP Server is unreachable, tell the developer to start the server and retry `/validate`.

## What validate_contract checks (16 total)
### Structural (1-12)
1. Section completeness — all 15 sections present with content
2. Topology consistency — every agent in §3 has a valid parent
3. Tool binding resolution — every tool in §3 STL Tool Bindings resolves
4. Pattern composition — agent tree follows adjacency rules
5. Infra module URLs — every module in §8 resolves via GitHub
6. NFR completeness — §10 has latency, throughput, availability
7. Business rule syntax — §9 IF/THEN rules parse correctly
8. Identity consistency — §15 agent names match §3 STL Agents
9. Agent Identity least-privilege — capabilities match tool bindings
10. Mermaid sync — §14 sequence matches §3 STL Sequence Summary
11. Skill version integrity — SHAs match catalog
12. Gateway route coverage — every MCP tool has a proxy route

### Diagram ↔ §3 STL Consistency (13-16)
13. Node parity — every agent in §3 STL Agents appears in diagram (and vice versa)
14. Edge parity — every flow in §3 STL Data Flows has a corresponding diagram edge
15. Hierarchy match — parent-child grouping in diagram matches §3 Parent column
16. A2A consistency — A2A delegation nodes in diagram match §3 STL A2A Delegations
