# AgentForge ‚Äî Design Prompt

You are calling the AgentForge Design Agent MCP Server. The server runs validate_spec (7-signal quality gate) as Step 0 before meta-skill reasoning begins. If validate_spec returns BLOCK, report the specific signal failures and stop ‚Äî do NOT proceed. If WARN, note the warnings and continue. If PASS, proceed normally. The server then produces a validated design contract with optional cosign attestation when enabled.

## Prerequisites
- Read `prompts/workspace.md` and resolve **APP_DIR**
- MCP Server configured (see IDE Setup Guide)
- `APP_DIR/spec.md` and `APP_DIR/plan.md` exist
- AGENTFORGE_API_KEY environment variable set (if required by your MCP server)

## Instructions
1. Read `APP_DIR/spec.md` and `APP_DIR/plan.md`
2. Call `design_start(spec, plan)` on the AgentForge Design Agent MCP Server
3. Receive taskId (< 2 seconds)
4. Poll `design_status(taskId)` every 3 seconds. Display progress to the developer:
   - "Loading meta-skills..." ‚Üí "Reasoning about patterns..." ‚Üí "Searching catalogs..."
   - "Querying Agent Registry..." ‚Üí "Writing ¬ß3 STL..." ‚Üí "Rendering diagram (per plan.md)..."
5. When status = "completed", call `design_result(taskId)`
6. Write returned files under **APP_DIR** only:
   - `APP_DIR/design-contract.md` (PRIMARY ‚Äî 15 sections with ¬ß3 Structured Topology Language)
   - If `diagram_format` is `eraserio`: `APP_DIR/diagrams/component-topology.eraser` from `diagram_xml`, plus `APP_DIR/diagrams/component-topology.png` from base64 `diagram_png`. Share `diagram_edit_url` with the developer if present.
   - If `diagram_format` is `drawio`: `APP_DIR/diagrams/component-topology.drawio.xml` from `diagram_xml` + `.png`
   - If `diagram_format` is `svg`: `APP_DIR/diagrams/component-topology.svg` from `diagram_xml` + `.png`
   - `APP_DIR/diagrams/hadr-lifecycle.<same-format>` when returned
   ‚öÝ Do NOT expect `design-contract.json` ‚Äî it is generated by `/validate`, not `/design`.
7. **HA/DR diagram (required when plan.md has DR strategy):**
   - Read `APP_DIR/plan.md` ‚Üí primary region, DR region, DR strategy (Backup & Restore / Pilot Light / Warm Standby)
   - The **HA/DR lifecycle diagram** (`hadr-lifecycle.*`) MUST reflect that strategy with **four scenarios**:
     1. Initial provisioning (primary + DR footprint per strategy)
     2. Component failure / in-region HA
     3. DR failover (traffic + data to DR region)
     4. DR failback (restore primary)
   - Populate design-contract **¬ß13 HA/DR Views** with strategy name, regions, and RTO/RPO notes
   - Component topology diagram stays in `component-topology.*`; HA/DR is a **separate** diagram file
8. Tell the developer: "Review APP_DIR/design-contract.md ‚Äî pay attention to ¬ß3 STL tables (agents, tool bindings, data flows, sequence), ¬ß5 tool bindings, ¬ß13 HA/DR (matches plan.md DR strategy), ¬ß15 Agent Identity. Edit .md and/or diagrams, then run /validate."

## MCP not configured
If the Design Agent MCP Server is unreachable, tell the developer to start the server (or check MCP settings) and retry `/design`. They may also manually author `APP_DIR/design-contract.md` using `templates/design-contract-template.md` as a fallback.
